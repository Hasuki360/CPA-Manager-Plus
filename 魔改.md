# CPA Manager Plus 魔改记录

## 目标

- 配置页的“额度处理”增加 Antigravity 版开关。
- Antigravity 额度用完时，也能像 Codex 一样临时停用对应登录文件，并在恢复时间后自动启用。
- Manager Server 管理员访问密码允许简单密码；当前后端只要求非空，本次补测试防止以后被改回强制 32 位。
- 请求监控实时表格增加列显示开关，按钮放在“失败”筛选旁边。
- 请求监控实时表格“用量”列在中文界面保留 `I/O/R/C` 标签，但数字单位从 `K/M` 改为 `万/亿`。
- 公益站模型监控与 HTTP 500 通道熔断迁移到 AI 提供商页，作为两个独立卡片管理。
- 配置页在 CPA 连接由环境变量锁定时，仍允许保存请求监控的采集间隔、批量大小等运行参数。

  - 只有对应 provider 的开关打开时，才把事件交给 quota cooldown worker（`filterQuotaCooldownEvents` 按 provider 过滤，HTTP 500 事件始终放行）。
  - 装配并托管 `charityModelMonitorWorker`：启动时 `Start`，运行时设置变更时 `UpdateConfig`。
  - 把运行时的 `HTTP500Cooldown*` 参数注入 `collector.RuntimeConfig`（`runtimeConfigWithHTTP500Settings`）。
  - 运行日志补充 `antigravityQuotaCooldown` / `charityModelMonitor` 状态。

- `apps/manager-server/internal/worker/rate_limit_auto_disable.go`
  - 原来只处理 `provider=codex` 的 `429 usage_limit_reached`。
  - 现已补充识别真实的 Antigravity/Gemini 额度耗尽返回格式（`RESOURCE_EXHAUSTED` / `QUOTA_EXHAUSTED`）。
  - 支持从 `quotaResetTimeStamp` 解析绝对恢复时间，或通过 `quotaResetDelay/retryDelay` (如 `39h49m8s`) 解析出准确的到期冷却时间。
  - Antigravity 429 后会优先通过管理端 `/v0/management/api-call` 主动查询套餐额度；若 5 小时限额或周限额剩余为 0，则按耗尽窗口里最晚的 `resetTime` 关闭账号，到期自动恢复。
  - 套餐查询失败或没有可用窗口时，回退旧逻辑：继续使用 429 错误体里的 `quotaResetTimeStamp/quotaResetDelay`。
  - HTTP 500 不再单次触发关闭渠道；改为复用 `usage_events`，按同一 AI 提供商 API key 的 `source_hash` 在统计窗口内累计失败次数。
  - HTTP 500 达到阈值后，通过该请求的 `source_hash` 与 `codex-api-key` / `claude-api-key` provider 的 `api-key` 做 SHA-256 精确匹配，只给对应 AI provider 写入 `excluded-models: ["*"]`；到期自动移除 `"*"` 恢复通道。
  - HTTP 500 卡片只管理 AI 提供商通道，不关闭 `auth-files` 账号文件；429 额度冷却仍使用账号文件关闭/恢复链路。

- `apps/manager-server/internal/worker/charity_model_monitor.go`
  - 公益站模型监控改为从 AI 提供商读取 `codex-api-key` / `claude-api-key` provider。
  - 通过 provider 的 `base-url` 自动匹配薄荷、君の的公益和 AnyRouter。
  - AI 提供商配置了“自定义模型（可选）”时，只检查这些自定义模型；全部不存在则关闭通道，任意一个存在则启动回来。
  - 没有自定义模型时，才回退检查 `gpt-*` / `claude-*`，并排除 `image` 模型。
  - AnyRouter 仍只同步 Codex 请求头，不拉 pricing、不控制通道开关。
  - Codex CLI 版本检测与旧 Python 版一致：请求 `https://registry.npmjs.org/@openai%2Fcodex/latest`，缓存 6 小时；失败时使用上次缓存，再失败则回退 `0.142.5`。

## 使用方式

### 本地 Docker 启动验证

构建时建议注入版本号，让面板“管理面板版本”显示为原版 tag + 魔改后缀：

```bash
# 例：v1.11.0-hasuki
VERSION="$(git describe --tags --abbrev=0 upstream/main)-hasuki"
```

当前正式运行版本：`v1.11.1-hasuki`（上游 `67182e1e` / main，含 Sub2API 导入与 codex-inspection 安全自动恢复）。

当前项目按 Docker-only 部署，不使用原生安装脚本。

当前本地目录结构：

```text
/www/234G/CPA-Manager-Plus/
├── 魔改/      # Hasuki360/CPA-Manager-Plus，实际部署源码（git）
├── 原版/      # seakee/CPA-Manager-Plus，干净上游参照
├── 发布镜像/  # 本机构建导出的镜像 tar
└── 自动化脚本/ # 巡检等脚本
```

本机验证时进入魔改目录，并用临时 override 覆盖管理员密钥，避免把固定密码写入仓库：

```bash
cat >/tmp/cpamp-local-override.yml <<'EOF'
services:
  cpa-manager-plus:
    image: hasuki/cpa-manager-plus:local
    environment:
      CPA_MANAGER_ADMIN_KEY: "cpamp_local_change_me"
EOF

cd /www/234G/CPA-Manager-Plus/魔改
docker compose -p cpa-manager-plus -f docker-compose.manager.yml -f /tmp/cpamp-local-override.yml up -d --build
```

访问：

```text
http://localhost:18317/management.html
```

健康检查：

```bash
curl -fsS http://127.0.0.1:18317/health
```

### VPS Docker-only 部署

VPS 只使用 Docker 模式：拉取仓库源码，本地构建镜像并启动容器。

```bash
git clone https://github.com/Hasuki360/CPA-Manager-Plus.git
cd CPA-Manager-Plus
export CPA_MANAGER_ADMIN_KEY='换成莲纪自己的管理密钥'
docker compose -p cpa-manager-plus -f docker-compose.manager.yml up -d --build
```

后续更新：

```bash
cd CPA-Manager-Plus
git pull --ff-only origin main
export CPA_MANAGER_ADMIN_KEY='换成莲纪自己的管理密钥'
docker compose -p cpa-manager-plus -f docker-compose.manager.yml up -d --build
```

### 后期自动任务同步更新

后期可以在 VPS 上做定时任务，逻辑固定为：

1. `git fetch origin upstream`
2. 如需同步原版，先在本地服务器处理 `upstream/main` 合并到魔改分支，解决冲突并测试后推送 `origin/main`
3. VPS 只拉 `origin/main`
4. `docker compose -p cpa-manager-plus -f docker-compose.manager.yml up -d --build`
5. 健康检查 `/health`，失败则输出日志并告警

VPS 自动任务不要直接合并 `upstream/main`，避免无人值守时把上游冲突或行为变化带进生产容器。

### AI 提供商页的通道健康配置

当前通道健康类魔改集中在 `AI 提供商` 页面顶部，分成两个独立卡片：

1. `公益站模型监控与通道自愈`
   - 从 AI 提供商读取 Codex / Claude provider。
   - 优先按“自定义模型（可选）”检查。
   - 通过 `excluded-models: ["*"]` 自动关闭/恢复通道。
   - 检查间隔可以在卡片最后的编辑区直接修改，顶部只保留状态信息。

2. `HTTP 500 通道熔断`
   - 使用 `宽松 / 标准 / 严格 / 自定义` 策略卡片。
   - 设置全局默认统计窗口、触发次数、关闭渠道时长。
   - 仍按同一认证文件 / 通道累计 HTTP 500，不跨通道合并计数。

账号处理策略页只保留开关类策略，不再放公益站 JSON 或 HTTP500 参数面板。

### Antigravity 额度冷却

可在配置页打开，也可以用环境变量锁定：

```yaml
environment:
  USAGE_ANTIGRAVITY_QUOTA_COOLDOWN_ENABLED: "true"
```

环境变量锁定后，UI 只能查看，不能修改。

Antigravity 发生 429 后，worker 会先读取当前认证文件的 `auth_index` 和 `project_id`，通过 CPAMP 管理端代理查询 Antigravity 套餐额度：

1. 如果 5 小时限额或周限额剩余为 0，按这些耗尽窗口里最晚的 `resetTime` 关闭账号。
2. 如果周限额为 0，即使 5 小时限额还有剩余，也会按周限刷新时间关闭。
3. 如果套餐查询失败，才回退读取 429 错误体里的 `quotaResetTimeStamp/quotaResetDelay`。

Codex 周限/月限逻辑本次不改。
